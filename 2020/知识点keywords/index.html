1.插件与热修复

滴滴的virtual Apk处理四大组件的原理：

* Activity：在宿主apk中提前占坑，然后通过“欺上瞒下”的方式绕过校验，启动插件APK里的Activity；

* Service：通过代理Service的方式去分发，VirtualAPK使用了两个代理Service，即LocalService和RemoteService。

* BroadcastReceiver：将静态广播转为动态广播。

* ContentProvider：通过一个代理Provider进行操作的分发。

2.应用保活

*监听广播：监听全局的静态广播，比如时间更新的广播、开机广播、解锁屏、网络状态、解锁加锁亮屏暗屏（3.1版本），高版本需要应用开机后运行一次才能监听这些系统广播，目前此方案失效。可以更换思路，做APP启动后的保活（监听广播启动保活的前台服务）；

*定时器、JobScheduler：假如应用被系统杀死，那么定时器则失效，此方案失效。JobService在5.0,5.1,6.0作用很大，7.0时候有一定影响（可以在电源管理中给APP授权）；

*双进程（NDK方式Fork子进程）、双Service守护：高版本已失效，5.0起系统回收策略改成进程组。双Service方案也改成了应用被杀，任何后台Service无法正常状态运行；

*提高Service优先级：只能一定程度上缓解Service被立马回收。

3.图片缓存框架（Fresco与Glide）

Glide的缓存是每个大小不同的尺寸的图片都会缓存一份，优先就是显示非常快，但是浪费内存。glide是基于picasso二次开发。支持gif动图。
Glide为activity和fragment都创建了一个无界面的fragment，在无界面的fragment中去监听生命周期的状态。

缓存：

  • ActiveResources 活跃的内存缓存，使用的是弱引用的方式
  • LruResourceCache 内存缓存
  • ResourceCacheGenerator:缓存的被加工过的原数据缓存（磁盘)
  • DataCacheGenerator:原文件
  • SourceGenerator:从网络加载数据

优点：
1.生命周期集成，根据Activity或者Fragment的生命周期管理图片加载请求
2.格式支持丰富，支持Gif、WebP等
3.高效的缓存策略，灵活（Picasso只会缓存原始尺寸的图片，Glide缓存的是多种规格），加载速度快且内存开销小（默认Bitmap格式的不同，使得内存开销是Picasso的一半）

Fresco三级缓存：

  • bitmap(已解码)缓存，内存中
  • EncodedImage(未解码)缓存，内存中
  • 磁盘缓存，磁盘中。其中磁盘缓存划分为三层：缓冲层（BufferedDiskCache）、缓存层（DiskStroageCache）、文件存储层（DefaultDiskStorage）
  • 其实严格来算还有一个网络层。这一层可以自由替换，默认使用HttpURLConnection，可以替换成okhttp。

Fresco优点：
1.完善的内存管理功能，减少图片对内存的占用，即便在低端机器上也有着不错的表现。
2.Ui层扩展极好，能自定义各种drawabel叠加在一起。自定义图片加载过程，（先低清晰图，再高清图），自定义绘制过程(圆角，占位，进度)
3.格式支持丰富，渐进式显示图片，支持gif和Webp
4.在低版本系统中使用匿名共享内存存储缓存，避免java的Gc和OOM。
5.适用于需要高性能加载大量图片的场景
6.头像和大图在磁盘上分开缓存，避免小图数量较多，触发回收
7.在DraweeView的onDetach释放，使用DeferredReleaser，延期释放资源,并且可以合并多次释放资源

实现图片缓存框架的思路：
异步加载：线程池与线程切换
缓存：LruCache、DiskLruCache（缓存key的定义）
防止OOM：软引用、onLowMemory、图片压缩、Bitmap像素存储位置
内存泄露：ImageView回收引起的生命周期管理
列表滑动加载的问题：加载错乱、滚动过程暂停加载、切换请求优先级

Bitmap的存储位置：
8.0 的Bitmap的分配在native堆：
创建native层Bitmap，在native堆申请内存。
通过JNI创建java层Bitmap对象，这个对象在java堆中分配内存。
7.0 像素内存的分配在java堆中：
通过JNI调用java层创建一个数组
然后创建native层Bitmap，把数组的地址传进去。

4.OkHttp

注意事项：
1.正在运行的异步任务runningAsyncCalls不得超过64个，同一个host下的请求不能超过5个
2.自己添加Accept-Encoding: gzip 居然结果会不解压，需要业务自己解压。BridgeInterceptor没有配置Accept-Encoding默认会gzip压缩的，同时也会解压response。
3.缓存使用了DiskLruCache方式，日志文件策略，记录2000个最近操作记录。

http读取缓存流程：

1.判断有缓存，检查缓存是否过期，没有过期直接使用
2.网络请求带上Etag或者Last-Modified,由服务端检查缓存是否过期，没有过期则会返回304，否则返回200和真实数据。

##5.Retrofit

原理：在retrofit.create(Target.class)使用JDK的动态代理创建代理类，在方法执行时解析方法上的注解和参数的注解获取请求参数，同时将请求结果转化成方法的返回参数的格式。具体就是使用HttpServiceMethod包装类保存各种参数，其中：
-ParameterHandler<?>[] parameterHandlers 注解解析器，负责解析 API 定义时每个方法的参数，并在构造 HTTP 请求时设置参数；
-okhttp3.Call.Factory callFactory：负责创建 HTTP 请求，HTTP 请求被抽象为了 okhttp3.Call 类，它表示一个已经准备好，可以随时执行的 HTTP 请求；
-Converter<ResponseBody, R> responseConverter：负责把服务器返回的数据（JSON、XML、二进制或者其他格式，由 ResponseBody 封装）转化为 R 类型的对象；
-CallAdapter<R, T> callAdapter：把 retrofit2.Call 转为 T（注意和 okhttp3.Call 区分开来，retrofit2.Call 表示的是对一个 Retrofit 方法的调用），这 个过程会发送一个 HTTP 请求，拿到服务器返回的数据（通过 okhttp3.Call 实现），并把数据转换为声明的 T 类型对象（通过 Converter<R, T> 实现）；

6.LeakCanary和BlockCanary

LeakCanary原理：将watch的对象生成加入唯一id加入Set集合，然后使用WeakReference引用该对象，同时为WeakReference对象指定一个ReferenceQueue对象，一旦WeakReference中指向的对象被回收，则该WeakReference就会被加入queue中。触发gc后然后检查queue中是否有该对象。

BlockCanary原理：在Looper.loop循环中，有一个Printer输出了各个关键节点。测试是否block只需要自定义一个Printer即可。

7.WMRouter

核心内容为ServiceLoader，记录了Map<Class, ServiceLoader>。@RouterService方式就是从这个map中取出注册的接口class，然后反射生成实例。

页面跳转的路由功能：
默认DefaultRootUriHandler，里面包含PageAnnotationHandler–>UriAnnotationHandler–>RegexAnnotationHandler–>StartUriHandler。
UriHandler中可以包含一个ChainedInterceptor。
UriAnnotationHandler包含map<String, PathHandler>，map的key为schema://host
PathHandler中也是包含map<String, UriHandler>，key是具体路径，UriHandler为具体的处理handler。PathHandler中的register（）方法根据传入的参数，生成具体的UriHandler（UriTargetTools.parse(target, exported, interceptors)）。
