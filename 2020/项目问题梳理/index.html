1.IM即时通讯

协议与数据格式

使用tcp协议保证消息的可靠性，数据的格式使用我们自主开发的AACE格式(集成了压缩和加密)，解析快、体积小、更安全。
消息对即时性要求高，并发量高，使用AACE加速解析速度对服务端和客户端的并发很有必要。

心跳策略

1.基于wifi和移动网络运营商设置心跳时间：wifi情况下4.5min，移动网络下3min20，其他网络下5min。时间设置太长的话会由于NAT超时（运营商）导致连接断开。
2.结合GCM(谷歌推送)，但是我们在国内使用不了。
3.智能心跳：前台时固定频率心跳,后台时自适应心跳（网络和失败成功统计情况）。

备注：NAT:简单的说运营商的网关需要维护一个外网IP、端口到内网IP、端口的对应关系，以确保内网的手机可以跟Internet的服务器通讯。

消息丢失

每条消息推送到客户端以后，客户端都会发送一个回执告诉服务端该消息已经收到。在推送断开期间，消息不能推送下来时，会先存在离线消息里面，当客户端打开以后，会立马去获取自己的离线消息。

耗电

长连接为了保活，心跳会持续发送网络请求，再加上为了保活的各种定时器加剧了耗电量。

为了尽量减少耗电，从以下方向考虑：
1.优化心跳策略，在网络不好或者请求失败时，智能延长心跳时间
2.将各种定时器方案改成监听广播的方式，因为定时器方案的保活在高版本中所起的作用越来越小，应用被杀，定时器也会失效
3.去掉双进程保活等无效方案；
4.使用第三方公共推送服务。

2.小程序方案

项目目标

1、和微信小程序一样提供丰富的客户端api，让开发者可以快速开发小程序；
2、后台提供拖拽式方式快速构建小程序。

整体实现原理：将后台上传的小程序下载在客户端，然后在客户端加载小程序。

版本管理

强制更新：在加载小程序时，检测到强制更新会立马下载最新，此时会展示加载等待页面；
静默更新：发现静默更新则会在依然加载旧版本的小程序，同时出发对新版小程序的加载；
提示更新：发现提示更新时则会弹出提示框，让用户选择是强制更新还是静默更新。

预加载：为了避免用户第一次打开小程序需要下载，我们在闲时会预加载高频小程序。

API管理

提供客户端层的api和小程序级别的api。

客户端api通过native://method?para=x的方式出发，主要提供一些小程序和客户端的交互，对小程序屏蔽客户端差异。实现方式：java调用h5：webView.loadUrl(js); h5调用客户端：shouldOverrideUrlLoading拦截请求url。使用注解的方式维护方法名和method的映射关系，同时还可以指定方法是否可以立即返回。

小程序api：1.提供小程序丰富的样式代码，小程序开发者可以快速编写ui样式 2.封装客户端api便于使用
小程序api可以大大的方便了开发者，同时将一些公用的库抽离出来，可以大大降低小程序体积大小。

安全

webview原生的安全漏洞：
1.addJavascriptInterface方法导致远程代码执行漏洞，前端可以获取到addJavascriptInterface的java对象的Class对象，Class.forName可以加载java.lang.Runtime 类,通过Runtime可以执行本地命令。该漏洞4.2版本后被修复，使用@JavascriptInterface进行注解从而避免漏洞攻击。
2.WebView默认开启密码保存功能，关键是密码会被明文存入 /data/data/com.package.name/databases/webview.db ,解决方案是关闭密码保存提醒
3.setAllowFileAccess允许小程序可以访问访问本地文件，容易导致本地隐私文件泄漏，setAllowUniversalAccessFromFileURLs设置是否允许通过 file url 加载的 Javascript 可以访问其他的源(包括http、https等源),setAllowFileAccessFromFileURLs设置是否允许通过 file url 加载的 Js代码读取其他的本地文件，小程序要求setAllowFileAccess(true)和setJavaScriptEnabled(true)让浏览器加载html和js代码，使用setAllowUniversalAccessFromFileURLs(false)和setAllowFileAccessFromFileURLs(false)禁止本地j访问文件。

客户端层面安全：
1.客户端api区分安全等级，有些方法需要授权才可以访问
2.客户端api白名单，有些api需要后台配置了才能访问
3.由于跨域问题，小程序的网络请求必须由客户端转发

性能

1.客户端调用h5使用evaluateJavascript，比loadurl速度快
2.缓存最近打开的5个小程序，保留每个小程序打开的完整Fragment栈，在栈中的小程序可以做到秒开

###客户端与小程序的相互调用
1.客户端调用小程序：loadUrl与evaluateJavascript
2.小程序调用客户端：addJavascriptInterface、shouldOverrideUrlLoading和onJsPrompt三种方式

3.搜索优化

基础原理

字典树(Trie树或者前缀树)：将字符串的每个字符作为节点插入到树中，从根节点到每个叶子节点代表每个字符串。Trie 树非常适合查找前缀匹配的字符串。

总共分为3个文件：
搜索字典树文件：存储搜索字典树的内容
结果字典树文件：将搜索树中的字符串倒序插入该字典树
业务关联结果文件：存储结果字典树中的叶子节点和业务id的关系。

TreeNode在内存中是存储在ArrayList，next、sub、parent为list中的index。
在硬盘上使用RandomAccessFile读写文件，每个文件可以理解成硬盘上的ArrayList结构，只不过头部4个字节会保存了这个ArrayList的长度。

建立索引步骤：

1.先将要搜索的字符串分隔插入搜索字典树中，例如ABCD分成ABCD、BCD、CD、D等字符串，然后都插入到搜索字典树中；
2.将搜索的字符串倒序插入结果字典树中，获取结果字典树结束节点的指针，例如上述的ABCD倒序为DCBA插入字典树中；
3.关联结果字典树中的叶子节点和业务id，例如上述字符串ABCD关联的是用户id为123的备注信息。

搜索步骤：

1.先在搜索字典树中搜索出符合条件的节点，然后取出该节点下所有的完整路径，
2.将上面获取完整路径倒序后，去结果字典树中搜索，搜索出符合条件的节点，同样取出该节点下所有的完整路径，
3.根据上面获取的完整路径的叶子节点的指针，去业务关联结果文件中匹配相关的业务id。

工程实现

如何优化文件的读写速度？

构建索引时，为了减少文件的读取操作，先批量操作内存，然后异步一起写文件。
为了搜索时减少文件的操作，将搜索字典树和结果字典树加载到内存中。

如何优化字典树的内存占用？

字典树常规结构是使用数组结构保存子节点，我们使用parent指向父节点，next指向子节点的第一个节点，sub指向兄弟节点。

为什么要引入结果字典树？

在不引入结果字典树时，由于每次插入一个字符串，需要插入n(字符串的长度)次，这必然会导致业务关联字典树的链表结构也要发生n次插入，（同样要删除该关联关系也需要n次删除），这个导致构建索引速度很慢。为了解决这个问题，引入了结果字典树，这样子，插入一个字符串只需要插入一个关联关系。

如何删除和修改搜索字符串与业务的关联关系

如果需要删除和修改，直接变更结果字典树和业务id的关联关系。

可优化的空间：

可以考虑对业务关联结果文件的链表改成红黑树，加快结果解析速度；
将字典树的兄弟节点链表优化成红黑树，加快字典树的遍历速度。
